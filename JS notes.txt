Hoisting: Hoisting in JavaScript is a behavior where declarations of variables, functions, and classes are conceptually moved to the top of their respective scopes during the compilation phase, before the code execution begins. 
This allows for the use of these declared entities before their actual appearance in the code. 
Hoisting is happens in var, let and const.

Temporal Dead Zone (TDZ): It refers to the time between the entering of a block and the point where a let or const variable is declared and initialized. 
Accessing a variable in this 'dead zone' throws a reference error, even though the variableis technically in scope.
During the TDZ the JS engine knowws the variable exists but you can't access it until the actual line of decaration is evaluated.

Shallow copy:
It copies only the top level of the object.A shallow copy creates a new object or array, but only copies the top-level properties. 
If any of these properties are references to other objects or arrays (i.e., nested data structures), only the references are copied, not the actual nested objects themselves. 
This means that both the original and the copied object/array will point to the same nested objects. 
We can even use slice() to make a shallow copy
Syntax: {...objName};     //spreadOperator
Ex:
    cconst array1 = ['a', 'b'];
    const array2 = ['c', 'd'];
    const combinedArray = [...array1, ...array2, 'e'];

console.log(combinedArray); // Output: ['a', 'b', 'c', 'd', 'e']

Deep copy:
A deep copy creates a completely independent copy of the original object or array, including all nested objects and arrays. 
This means that a new instance is created for every level of the data structure, ensuring no shared references between the original and the copy

Method to achieve deep copy

Promise:
Promise is an object that represents the eventual completion (or failure) of an asynchronous operation and its resulting value. It acts as a placeholder for a value that may not be available yet but will be resolved or rejected in the future. 

States of a Promise:
A Promise can be in one of three states:
- Pending: The initial state, where the asynchronous operation is still in progress and has not yet completed or failed.
- Fulfilled (Resolved): The operation completed successfully, and the Promise now holds the resulting value.
- Rejected: The operation failed, and the Promise holds the reason for the failure (an error).

Consuming Promises:
You can consume the result of a Promise using the .then() and .catch() methods:
- .then(onFulfilled, onRejected): This method allows you to attach callbacks for when the Promise is fulfilled or rejected.
    onFulfilled: A function that executes if the Promise is fulfilled, receiving the resolved value as an argument.
    onRejected: An optional function that executes if the Promise is rejected, receiving the rejection reason as an argument.
- .catch(onRejected): This is a shorthand for .then(null, onRejected) and is commonly used for handling errors (rejections) at the end of a Promise chain.

Async and Await:
Async and Await in JavaScript are used to simplify handling asynchronous operations using promises. By enabling asynchronous code to appear synchronous, they enhance code readability and make it easier to manage complex asynchronous flows.

Async Function
The async function allows us to write promise-based code as if it were synchronous. This ensures that the execution thread is not blocked. Async functions always return a promise. If a value is returned that is not a promise, JavaScript automatically wraps it in a resolved promise.
Syntax:
async function myFunction() {
  return "Hello";
}

Await Keyword
The await keyword is used to wait for a promise to resolve. It can only be used within an async block. Await makes the code wait until the promise returns a result, allowing for cleaner and more manageable asynchronous code.

Error Handling in Async/Await
JavaScript provides predefined arguments for handling promises: resolve and reject.

resolve: Used when an asynchronous task is completed successfully.
reject: Used when an asynchronous task fails, providing the reason for failure.2
